# codex

> 项目法典。沉淀做完一件事之后真正值得留下来的东西：决策、拓扑、契约、坑、SOP。
>
> 本目录是接手人第一份要读的资料，也是自己回头查"当时为什么这么干"的第一去处。
> 代码本身能说清的事，不在这里复述；这里只写代码不会告诉你的"为什么"。

## 触发协议 — 给 AI 助手

> **当用户的指令包含「读 codex/README.md」或类似表达，并要求"完成自动化文档落实流程 / 沉淀 / 复盘"时，按以下 6 阶段执行**。
> 用户在指令尾部会给出主题描述（如"最近做了 X 功能"、"修复了 Y 难点"、"阶段性复盘"）。

### 阶段 1 — 解析

- 从指令尾部提取**主题**（场景 + 范围）。常见三种：
  - **功能型**："最近做了 X"、"X 功能落地"  → 大概率新增 Domain
  - **修复型**："修了 Y bug"、"解决 Z 问题"  → 多入 SCRATCH，跨多文件则升 Domain
  - **复盘型**："阶段性复盘"、"近 N 天"  → 综合性，可能产出多条 SCRATCH + 0~1 个 Domain
- 若指令未给主题，**用 `git log --since=...` 自动推断**：复盘默认覆盖最近 7 天，功能/修复默认覆盖最近 3 条提交。

### 阶段 2 — 诊断

并行收集证据，不要凭印象：

```bash
git log --oneline --since="<范围>"        # 提交概览
git diff <range> --stat                    # 改动文件清单
git diff <range> -- <关键文件>            # 关键改动细节
```

把每一条想写进文档的事实，先在 `git log` / `git diff` / `grep` 找到对应行号 / 文件 / 提交哈希作为锚点。**没有锚点的"经验"不写**，先丢 SCRATCH 让它发酵。

### 阶段 3 — 决策（写哪儿）

按 [§ 晋升标准](#晋升标准) 判断：

| 判断 | 去处 | 文件 |
|---|---|---|
| 跨 ≥2 个文件 且 有外部依赖（CI / 密钥 / 服务） | 正式 Domain | `codex/NN-<域名>.md` 新文件 |
| 跨 ≥2 个文件 但 无外部依赖，或 单文件改动但已踩过坑 | 正式 Domain | 同上 |
| 单文件、孤立、未踩坑 | SCRATCH | `codex/scratch.md` 追加条目 |
| 既有 Domain 的增量更新（如 §01 已存在，本次只是补一条经验） | 既有文件 | 编辑对应 Domain 文件，更新 README 索引中"上次更新"日期 |

### 阶段 4 — 撰写

**Domain 章节** — 套用 [§ Domain 模板](#domain-模板)，必填四件套 `.1 / .2 / .4 / .9`：

- `.1` 一句话总结（不超过两行）
- `.2` 决策与权衡（表格对比 + 不变量）
- `.4` 关键文件 & 行号（带 `path:line` 锚点）
- `.9` 经验教训（具体到能避坑的颗粒度，不写"要小心"这种空话）

**SCRATCH 条目** — 三行格式：

```markdown
### [YYYY-MM-DD] <一句话主题>
- **现象**: <看到了什么、什么时候、什么场景>
- **当前判断**: <初步怀疑是什么 / 临时绕过方式>
- **遗留疑问**: <还需验证的事 / 未来若升 Domain 该补什么>
```

### 阶段 5 — 校对（不要跳）

写完后逐项核查，**任何一条对不上都要修，不要发明**：

1. 每个 `path:line` 引用：用 `grep -n` 或 `Read` 验证文件存在、行号附近确实是引用的符号
2. 每个 i18n key、配置项、命令名：grep 验证真实存在
3. ASCII 图：每个箭头两端的盒子在拓扑上确实有关系（不要拍脑袋画）
4. 失败模式表的每条"现象→根因→修复"：现象必须是真实见过的（git log / 对话 / SCRATCH 有记录），不要写假想故障

### 阶段 6 — 写入 + 报告

**写入：**
- 新 Domain：创建 `codex/NN-<域名>.md` 文件 + 在本 README 的[索引表](#索引)追加一行
- 既有 Domain 增量：编辑对应文件，更新 README 索引"上次更新"日期
- SCRATCH 新条目：在 `codex/scratch.md` 最末追加（按时间倒序也行，但保持一致）

**报告**（2-3 句给用户）：
- 写到哪个文件、哪个章节
- 校验过多少处代码引用
- 是否有发酵中的 SCRATCH 条目值得关注

## 写作约定

### 状态标签

- ✅ 已落地并验证
- 🚧 进行中 / 半成品
- ⚠️ 已知问题 / 待迁移
- 💡 想法 / 未启动

### 引用规范

- 代码引用：`path/from/repo/root.ext:line`（如 `src/components/SettingsView.tsx:322`）
- 行号是**当时快照**，重构后会漂移；优先按符号名定位，行号只做导航辅助
- 外部链接尽量贴 commit hash 永链；上游官方文档可直接贴 URL
- 跨章节引用用文件相对路径 + 锚点（如 `[§ 01.4](./01-updater.md#014-关键文件--行号)`）

### Domain 模板

```markdown
# NN <Domain 名> (<英文代号>)

**状态**: <标签>  **首次落地**: [YYYY-MM-DD]  **最近更新**: [YYYY-MM-DD]  **域**: <一句话定位>

## NN.1 一句话总结
## NN.2 决策与权衡          — 为什么这么做，否决了哪些方案
## NN.3 实现拓扑            — 端到端数据流 / 控制流，ASCII 图优先
## NN.4 关键文件 & 行号      — 全局索引表，重构时优先更新
## NN.5 数据契约            — JSON/Proto schema、API contract
## NN.6 状态机 / 生命周期    — 如适用
## NN.7 失败模式与排查       — 现象 → 根因 → 修复 三列表
## NN.8 SOP                — 上手 / 发布 / 回滚的逐步操作
## NN.9 经验教训            — 这次踩过的、下次别再踩的
## NN.10 未来扩展           — 已识别但未做的，含可放弃理由
```

不是每节都要填。空节可省略，但 `.1 / .2 / .4 / .9` 是必填四件套。

### 晋升标准

`scratch.md` 里的条目，满足以下任一条件就提升为正式 Domain：

- 跨过 ≥2 个文件
- 有外部依赖（CI / 密钥 / 服务 / 第三方 API）
- 至少踩过 1 次坑并修复

晋升时：在 `codex/` 新建 `NN-<域名>.md`，把 SCRATCH 条目作为 `.9 经验教训` 第 1 条，删掉 `scratch.md` 里的对应条目，更新 README 索引。

### codex vs 代码注释 的边界

| 内容 | 去处 |
|---|---|
| "为什么选 A 不选 B" | codex |
| "线上踩过的具体坑" | codex |
| "上手要做的 N 步" | codex |
| "这个 hack 是临时的"的原因 | codex（代码内 `// TODO` 留指针） |
| "代码这一行在干啥" | 不写注释，让代码自己说话 |

## 触发示例

下面这些指令都会触发 [§ 触发协议](#触发协议--给-ai-助手)：

```
读 codex/README.md，完成最近自动更新功能的文档落实

读 codex/README.md，完成阶段性复盘（最近 7 天）

读 codex/README.md，沉淀刚才修复的 i18n 占位符 bug

/codex 国际化体系       # 如已配置 slash command
```

也接受**省略主题**的形式，会自动从 git log 推断：

```
读 codex/README.md，沉淀最近的工作
```
